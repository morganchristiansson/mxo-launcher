## Current status

This crash is now best treated as a **historical forced-runtime-after-failed-init reference**, not the best description of the current clean binder/scaffold runtime path.

## Why this is different from original `launcher.exe`

`../../launcher.exe` is the source of truth.

From static analysis, original launcher startup does **more work before runtime**:

1. builds launcher object at `0x4d6304`
2. performs pre-client environment setup at `0x402ec0`
3. resolves `ILTLoginMediator.Default` into `0x4d2c58`
4. populates arg7 selection state from `CLauncher+0xa8/+0xac`
5. passes all of that into `InitClientDLL`
6. only then calls `RunClientDLL`

Our fresh dump differs from original launcher behavior in the most direct possible way:

- `InitClientDLL` already returned **`-7`**
- but we forced `RunClientDLL` anyway for diagnostics

So dump `73` should not be read as "original launcher would crash here too".
It should be read as:

- if launcher-owned prerequisites are missing,
- and runtime is forced anyway,
- `client.dll` later dereferences a null subordinate state object at `0x62999968+0x04`.

## What the crash proves

### Proven
- the current project-local launcher can reproduce the classic `0x623b3573` crash on demand
- `cres.dll` is present in the fresh current dump
- the crash survives after adopting the original 8-argument `InitClientDLL` frame shape
- the remaining problem is not just DLL load order

### Not proven
- that the correct fix is to inject `client.dll` memory
- that `SetMasterDatabase` alone is sufficient
- that arg7 is version info

## Best current conclusion

This crash remains best explained by **missing launcher-owned startup state** rather than by a need for ad-hoc internal patching.
The highest-value missing pieces remain:

- `0x4d6304`
- `0x4d2c58`
- arg7 selection state from `CLauncher+0xa8/+0xac`
- pre-client setup around `0x402ec0`
