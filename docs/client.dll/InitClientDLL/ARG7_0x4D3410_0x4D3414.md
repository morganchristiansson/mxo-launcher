# InitClientDLL arg7 source: `CLauncher+0xa8/+0xac`

## Correction to earlier labeling

The earlier label:

- `packedVersionInfo`

is now considered too speculative and likely misleading.

## What is actually known

At the `InitClientDLL` call site:

```asm
mov eax, [ebx+0xac]
mov ecx, [ebx+0xa8]
and eax, 0x00ffffff
shl ecx, 0x18
or  eax, ecx
push eax        ; arg7
```

The `this` object here is the global launcher object at:

- `0x4d3368` (`CLauncher`)

So these fields are concretely:

- `[ebx+0xa8]` = `0x4d3410`
- `[ebx+0xac]` = `0x4d3414`

## New high-confidence finding

The writer for these two fields is the path around `0x40d763..0x40d810`, which ends with:

```asm
mov [0x4d3410], edi
mov [0x4d3414], ebx
```

That same path also touches the registry persistence slot for the saved world name, but current disassembly now points to a **write-back**, not a read:

- key root: `HKEY_LOCAL_MACHINE`
- subkey: `Software\Monolith Productions\The Matrix Online\`
- value name: `Last_WorldName`
- current best import interpretation of the calls at `0x40d7c4..0x40d802` is:
  - `RegOpenKeyExA`
  - `RegSetValueExA`
  - `RegCloseKey`

So by the time this path writes `Last_WorldName`, the launcher has already resolved the selected world/service state through earlier launcher-owned logic.

## Why this matters

This strongly suggests arg7 is **not simply launcher version/build info**.
It also tightens an important correction:

- `Last_WorldName` now looks more like a persistence/output artifact of the selection path,
- not a complete explanation of the selection path by itself.

So arg7 appears to be derived from launcher-side world/server selection state, with `Last_WorldName` acting as one saved-world persistence hook rather than the whole derivation.

## Current best interpretation

Arg7 is a **packed launcher selection / world-identification value** derived from `CLauncher+0xa8/+0xac`.
Current evidence now points to a broader launcher-owned selection-resolution chain here:

- `0x48baea` produces a value that is immediately split into:
  - low 16 bits -> zero-extended into `ebx` -> written to `0x4d3414`
  - high 16 bits -> shifted into `edi` / sign-extended -> written to `0x4d3410`
- launcher state rooted at `0x4d3584` is then consulted through methods such as `+0xfc`, `+0x100`, and `+0xe4`,
- and only after that does the launcher persist the chosen world name back to `Last_WorldName`.

A new static clarification materially narrows what `0x4d3584` is.

The initializer at `launcher.exe:0x496480..0x496491` calls the same wrapper ctor `0x4030d0` with:

- interface string `0x4ab34c = "ILTLoginMediator.Default"`
- output slot `0x4d3584`

That means `0x4d3584` is **not just an unknown generic object root**.
It is another launcher-global runtime pointer slot registered under the same `ILTLoginMediator.Default` service name already known for `0x4d2c58`.

So the current best interpretation is tighter than before:

- arg7 selection resolution appears to use a **sibling `ILTLoginMediator.Default`-style slot at `0x4d3584`**,
- and the launcher queries that sibling slot through `+0xfc`, `+0x100`, and `+0xe4` while validating/splitting the packed result from `0x48baea`.

Exact semantic names are still open.

## What is not yet proven

- the exact meaning of the high 8 bits sourced from `CLauncher+0xa8`
- the exact meaning of the low 24 bits sourced from `CLauncher+0xac` (current writer path makes that look more like a low-16-bit id with zero upper bits than a fully used 24-bit field, but this is not yet proven globally)
- whether the value is a world id, server id, shard selection, or another packed selector
- what `0x48baea` returns before the final `edi` / `ebx` writes
- what concrete object/interface is rooted at `0x4d3584`
- how that `0x4d3584` service chain (`+0xfc`, `+0x100`, `+0xe4`) validates or transforms the selected world before `0x4d3410` / `0x4d3414` are written

## Updated practical note

Current launcher scaffolding now rebuilds arg7 with the exact original bit packing rule and can override either:
- packed arg7 directly, or
- split `CLauncher+0xa8` / `+0xac` placeholders.

The current mediator probe now also carries the **low 24 bits** of that packed arg7 value into the fake selection-descriptor id field, instead of reusing the trivial selection index there.
That is a better diagnostic fit for the observed client `%s_%X` path formatting, but current evidence still says the real fix is to recover the original launcher-owned selection-resolution chain rather than guessing stable hardcoded selector values or treating `Last_WorldName` alone as the derivation.

## New diagnostic progress: stricter sibling-slot modeling and a non-zero arg7 probe

The replacement launcher now uses a slightly stricter `0x4d3584`-style mediator model during arg7 reconstruction work.

Current diagnostic behavior:
- `+0x3c` now returns the configured selected world id instead of always `0`
- `+0xfc` / `+0x100` / `+0xe4` / `+0x40` no longer generically accept every in-range index
- they now exact-match the currently configured selected world / variant pair
- optional diagnostic knobs now exist for shape experiments:
  - `MXO_MEDIATOR_WORLD_TYPE`
  - `MXO_MEDIATOR_VARIANT_STATE`

This is still not faithful original reconstruction, but it is less misleading than treating every in-range value as equivalent.

### New non-zero arg7 result

Representative run:
- `MXO_ARG7_SELECTION=0x0500002a`
- `MXO_MEDIATOR_SELECTION_NAME=Vector`
- `make run_binder_both`
- representative dump from that run family:
  - `~/MxO_7.6005/MatrixOnline_0.0_crash_59.dmp`
  - `EIP=0x003e5e8a`
  - crash-time stack still begins with stale startup-frame values

Observed launcher-side sibling-slot phase:
- `+0xfc(worldIndex=0x2a)` -> `"Vector"`
- `+0x100(worldIndex=0x2a)` -> `1`
- `+0xe4(variantIndex=0x05)` -> `0`
- launcher-side arg7 rebuild therefore still succeeds as:
  - `a8=0x00000005`
  - `ac=0x0000002a`
  - `packed=0x0500002a`

But the later client-side `+0x40` requests on that same run were:
- `selectionIndex=0x05000005`

not:
- `0x0000002a`
- and not the untouched packed value `0x0500002a`

A fresh static pass now explains where that value comes from.
Inside `client.dll:0x62170dc1..0x62170e59` the function:

```asm
mov ebx, esi
and esi, 0x00ffffff       ; low-24-bit world/selection side
shr ebx, 0x18             ; high-8-bit variant/type side in bl
...
mov [ebp+0x14], bl        ; overwrite only the low byte of the original arg7 stack slot
...
mov esi, [ebp+0x14]       ; later reload the mutated dword
```

So with original packed arg7:
- `0x0500002a`

that scratch rewrite produces:
- `0x05000005`

because the original stack slot is reused as a temporary dword whose:
- high byte still comes from original arg7 high8 (`0x05`)
- low byte is overwritten with that same high8 (`0x05`)
- middle bytes remain whatever survived from the original low-24-bit field

Important additional detail from the same block:
- before doing that scratch rewrite, the client also stores the original low-24-bit selection id separately via:

```asm
push esi                    ; esi already masked to low 24 bits
mov  ecx, 0x629e1c7c
call 0x620011e0
```

So the client is not simply "losing" the original low-24-bit selection id.
It appears to preserve that stable value through nearby state rooted at `0x629e1c7c`, then later uses the mutated arg7 stack slot only as the lookup key shape for `arg6->+0x40`.

## New clarification: `0x629e1c7c` is client-side `CreateCharacterWorldIndex`

A closer static pass now identifies that preserved low-24-bit sink much more concretely.

Static initializer:

```asm
627c5320: push 0
627c5322: push 0x62868a50      ; "No Description."
627c5327: push 0
627c5329: push 0x62870f04      ; "CreateCharacterWorldIndex"
627c532e: mov  ecx, 0x629e1c7c
627c5333: call 0x620022f0
```

This constructs a client-side console-int object at `0x629e1c7c` named:
- `CreateCharacterWorldIndex`

The setter used from `InitClientDLL` is the generic console-int store at `0x620011e0`.
The constructor / base path also matter:
- `0x620022f0` seeds default/current to `0`
- base console-var ctor `0x622a2270` zeroes callback slots at `+0x20 / +0x24 / +0x28`

That means the current best interpretation of the `InitClientDLL` write is now:
- masked arg7 low-24-bit selection id is stored into client-owned console/config state,
- specifically the console-int `CreateCharacterWorldIndex`,
- before later variant checks and before the `+0xec` selection/config handoff is built.

Current direct xref result is still narrow:
- one concrete direct read of its current value field `0x629e1cb0` appears at `client.dll:0x62054cbd`
- surrounding strings in that broader path include:
  - `CharCreate_2_Finish`
  - `CharCreate_2_Back`
  - `Loading Character`

Initial interpretation was that this looked more like later character-creation state than the immediate crash path.
But that is now too narrow.
Current runtime observation is that the patched-client crash is visibly happening during the in-game **Loading Character** phase (loading bar and status text already on screen).
So this `CreateCharacterWorldIndex` path should stay on the active suspect list as a promising later consumer of the persisted arg7 low-24-bit state.

That means the later client-side `+0x40` request is **not arbitrary**.
It is a mutated scratch form derived from the original packed arg7 stack slot.

A second static anchor now clarifies what the client expects back from that call.
Inside `client.dll:0x62195ff0`, the `%s_%X` profile-path formatter:
- calls `arg6->+0x40(selectionIndex)` with that scratch-shaped request,
- then follows the returned descriptor through `descriptor->+0x10`,
- and reads payload fields at:
  - `+3` -> name pointer
  - `+7` -> selection id

So the current best interpretation is:
- the scratch-shaped `+0x40` request is only the *lookup key shape*,
- while the descriptor result still needs to carry the stable world/selection name and stable low-24-bit selection id back out for path building.

What is still not fully proven is the full semantic contract of that mutated request.
The replacement launcher now accepts this scratch-shaped `+0x40` request diagnostically and returns the configured world/selection descriptor anyway, but that correction still did **not** move the late crash family.
So the shape is now better understood, while the full expected selection semantics are still open.

## Reimplementation implication

Until this field is understood better, the custom launcher should avoid claiming arg7 is a version field.
A neutral name like `packedArg7Selection` is safer.

Also, the current scaffold should treat the sibling `0x4d3584` path as a **selection-resolution probe**, not a solved derivation:
- launcher-side `+0xfc / +0x100 / +0xe4` behavior is now at least being exercised with non-zero values,
- the later client-side `+0x40` request format is now partially explained as a mutated arg7 stack-scratch value,
- but the full semantic contract behind that scratch-shaped `+0x40` request still does not appear solved because accepting it diagnostically still does **not** move the late crash.
