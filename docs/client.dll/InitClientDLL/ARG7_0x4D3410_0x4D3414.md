# InitClientDLL arg7 source: `CLauncher+0xa8/+0xac`

## Correction to earlier labeling

The earlier label:

- `packedVersionInfo`

is now considered too speculative and likely misleading.

## What is actually known

At the `InitClientDLL` call site:

```asm
mov eax, [ebx+0xac]
mov ecx, [ebx+0xa8]
and eax, 0x00ffffff
shl ecx, 0x18
or  eax, ecx
push eax        ; arg7
```

The `this` object here is the global launcher object at:

- `0x4d3368` (`CLauncher`)

So these fields are concretely:

- `[ebx+0xa8]` = `0x4d3410`
- `[ebx+0xac]` = `0x4d3414`

## New high-confidence finding

The writer for these two fields is the path around `0x40d763..0x40d810`, which ends with:

```asm
mov [0x4d3410], edi
mov [0x4d3414], ebx
```

That same path also touches the registry persistence slot for the saved world name, but current disassembly now points to a **write-back**, not a read:

- key root: `HKEY_LOCAL_MACHINE`
- subkey: `Software\Monolith Productions\The Matrix Online\`
- value name: `Last_WorldName`
- current best import interpretation of the calls at `0x40d7c4..0x40d802` is:
  - `RegOpenKeyExA`
  - `RegSetValueExA`
  - `RegCloseKey`

So by the time this path writes `Last_WorldName`, the launcher has already resolved the selected world/service state through earlier launcher-owned logic.

## Why this matters

This strongly suggests arg7 is **not simply launcher version/build info**.
It also tightens an important correction:

- `Last_WorldName` now looks more like a persistence/output artifact of the selection path,
- not a complete explanation of the selection path by itself.

So arg7 appears to be derived from launcher-side world/server selection state, with `Last_WorldName` acting as one saved-world persistence hook rather than the whole derivation.

## Current best interpretation

Arg7 is a **packed launcher selection / world-identification value** derived from `CLauncher+0xa8/+0xac`.
Current evidence now points to a broader launcher-owned selection-resolution chain here:

- `0x48baea` produces the packed source split into the later `edi` / `ebx` writes,
- launcher object/service state rooted at `0x4d3584` is then consulted through methods such as `+0xfc`, `+0x100`, and `+0xe4`,
- and only after that does the launcher persist the chosen world name back to `Last_WorldName`.

Exact semantic names are still open.

## What is not yet proven

- the exact meaning of the high 8 bits sourced from `CLauncher+0xa8`
- the exact meaning of the low 24 bits sourced from `CLauncher+0xac`
- whether the value is a world id, server id, shard selection, or another packed selector
- what `0x48baea` returns before the final `edi` / `ebx` writes
- what concrete object/interface is rooted at `0x4d3584`
- how that `0x4d3584` service chain (`+0xfc`, `+0x100`, `+0xe4`) validates or transforms the selected world before `0x4d3410` / `0x4d3414` are written

## Updated practical note

Current launcher scaffolding now rebuilds arg7 with the exact original bit packing rule and can override either:
- packed arg7 directly, or
- split `CLauncher+0xa8` / `+0xac` placeholders.

That is useful for differential probing, but current evidence still says the real fix is to recover the original launcher-owned selection-resolution chain rather than guessing stable hardcoded selector values or treating `Last_WorldName` alone as the derivation.

## Reimplementation implication

Until this field is understood better, the custom launcher should avoid claiming arg7 is a version field.
A neutral name like `packedArg7Selection` is safer.
